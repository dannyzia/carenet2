## 1. State Transition Validation

### 1.1 Channel Partner Status Lifecycle

**Valid transitions** (per spec 2.3 and 4.1):
- `pending_approval` Ôćĺ `active` (admin approval)
- `pending_approval` Ôćĺ `rejected` (admin rejection)
- `active` Ôćĺ `suspended` (admin suspension)
- `active` Ôćĺ `deactivated` (admin deactivation)
- `suspended` Ôćĺ `active` (admin reinstatement)
- `rejected` Ôćĺ `pending_approval` (reapplication)
- `deactivated` Ôćĺ `active`? (not specified; assume admin can reactivate)

**Constraints needed**:
- Database CHECK constraint on `channel_partners.status` to allow only defined values.
- Application logic must enforce transitions; consider adding a state machine guard in service layer.
- `reapplication_count` increments on `rejected` Ôćĺ `pending_approval`.

**Validation points**:
- Ensure `referral_code` generated only on `pending_approval` Ôćĺ `active` transition.
- `approved_by`, `approved_at` set on approval; `rejected_by`, `rejected_at`, `rejection_reason` on rejection.
- `suspended_by`, `suspended_at`, `suspended_reason` on suspension.
- `deactivated_by`, `deactivated_at` on deactivation.

### 1.2 Commission Status Lifecycle

**Valid transitions** (per spec 4.4 and plan 4.2):
- `pending` Ôćĺ `credited` (invoice payment verified/paid)
- `credited` Ôćĺ `paid` (ChanP confirms receipt)
- `pending` Ôćĺ `reversed` (invoice cancelled/disputed before payment)
- `credited` Ôćĺ `reversed` (invoice cancelled/disputed after payment)

**Constraints needed**:
- Database CHECK constraint on `channel_partner_commissions.status` (`pending`, `credited`, `paid`, `reversed`).
- Ensure `credited` Ôćĺ `paid` transition only allowed for ChanP owning the commission (RLS policy).
- `pending` Ôćĺ `credited` and `credited` Ôćĺ `reversed` only via SECURITY DEFINER functions.

**Idempotency**:
- `pending` Ôćĺ `credited` must check `status = 'pending'` before update.
- `credited` Ôćĺ `paid` can be repeated safely (no side effects).

**Audit logging**:
- Each transition must log corresponding CP_* audit event.

### 1.3 Rate Record Lifecycle

**Active rate definition**: `effective_to IS NULL`.
- New rate insertion triggers `close_previous_rate()` setting `effective_to = now()` on previous active record.
- `expires_at` is advisory; rate remains active after expiry until replaced.
- `expiry_notified` reset to `false` when `expires_at` extended (renewal).

**Transitions**:
- Insert new rate Ôćĺ previous rate closed.
- Update `expires_at` (renewal) Ôćĺ reset `expiry_notified = false`.
- Update `expiry_notified = true` when notification sent.

**Constraints**:
- UNIQUE PARTIAL index on (`channel_partner_id`, `lead_role`) WHERE `effective_to IS NULL` ensures at most one active rate per role.
- `effective_from` defaults to `now()`; `effective_to` nullable.

### 1.4 Lead Attribution Lifecycle

**Attribution methods**: `referral_code`, `cp_created`, `admin_assignment`.
- Once attributed, lead cannot be reassigned except by admin action (audit logged).
- `is_active` boolean; `deactivated_at`, `deactivation_reason` on deactivation.
- `registration_completed_at` set when proxyÔÇĹcreated lead activates account.

**Transitions**:
- `registration_completed_at` null Ôćĺ timestamp.
- `is_active` true Ôćĺ false (deactivation).

## 2. Idempotency Logic Validation

### 2.1 Commission Calculation Trigger

**Idempotency guarantee**: `UNIQUE(invoice_id, channel_partner_id)` constraint with `ON CONFLICT DO NOTHING` in `calculate_cp_commission()`.

**Validation**:
- Duplicate invoice insert (same `invoice_id`) Ôćĺ trigger fires again Ôćĺ `ON CONFLICT DO NOTHING` silently ignores.
- Should also check if commission already exists before insert (defensive).
- Ensure trigger function does not raise exception on conflict (must handle gracefully).

### 2.2 Commission Credit Trigger

**Idempotency**: Check `status = 'pending'` before updating to `credited`.
- If status already `credited` or `reversed`, no-op.
- Use `UPDATE ... WHERE status = 'pending'`; if zero rows updated, log warning.

**Validation**:
- Multiple payment status updates (e.g., `verified` Ôćĺ `paid`) should not cause double credit.
- Edge: invoice status changes from `verified` back to `unpaid`? Not defined; commission should remain `credited` (payment already verified). Need rule.

### 2.3 Rate Closure Function

**Idempotency**: `close_previous_rate()` must be safe to call multiple times.
- Should only update rows where `effective_to IS NULL`.
- If no active rate exists, no-op.

**Race condition**: Concurrent rate updates could cause two active rates if not serialized.
- Use `SELECT ... FOR UPDATE` on the active row before closing.
- Or rely on UNIQUE partial index constraint which will raise error on second insert; function must handle error and retry.

**Recommendation**: Implement `close_previous_rate` with `FOR UPDATE SKIP LOCKED` and retry loop.

### 2.4 Referral Code Attribution

**Idempotency**: `attribute_lead_via_referral_code` uses `ON CONFLICT (lead_user_id) DO NOTHING`.
- If lead already attributed to same or different ChanP, no change.
- Conflict should be logged (info).

## 4. Rule Contradictions Analysis

### 4.1 Perpetual Commission vs Rate Expiry

**Rule**: ÔÇťCommission earned on every invoice generated by a lead indefinitely, as long as a rate is configured for that lead role.ÔÇŁ (rule 1)

**Rule**: ÔÇťEach perÔÇĹrole rate has a mandatory expires_at date. Rate stays active after expiry until renewed or replaced.ÔÇŁ (rule 5)

**Contradiction?** No. Expiry is advisory; rate remains active after expiry. Commission continues indefinitely as long as a rate exists (even expired). Implementation matches.

### 4.2 Commission Basis: Platform Commission vs Invoice Total

**Rule**: ÔÇťCalculated on the platform commission amount, not the invoice total.ÔÇŁ (rule 2)

**Implementation**: `platform_commission_amount` column stored; calculation uses `platform_commission_amount * (cp_commission_rate / 100)`.

**Check**: Must ensure `platform_commission_amount` is populated from invoiceÔÇÖs `platform_fee` column. The spec confirms `public.invoices` has `platform_fee` column.

### 4.3 Rate Changes Apply to Future Invoices Only

**Rule**: ÔÇťChanges apply to future invoices only ÔÇô never retroactive.ÔÇŁ (rule 4)

**Implementation**: Rate snapshot `rate_record_id` stored with each commission; new rate inserts new record with `effective_from = now()`. Existing commissions unaffected.

**Validation**: Ensure `calculate_cp_commission` uses rate where `effective_from <= invoice_generated_at` and `effective_to IS NULL` (or `effective_to > invoice_generated_at`). Actually, they use active rate at invoice time (effective_to IS NULL). ThatÔÇÖs correct.

### 4.4 Suspended/Deactivated ChanP ÔÇô No New Commissions

**Rule**: ÔÇťNo NEW commission entries are created for invoices generated after the suspension/deactivation date.ÔÇŁ (rule 11)

**Implementation**: `calculate_cp_commission` must check `channel_partners.status` at invoice creation time. If status not `active`, skip.

**Edge**: Suspension timestamp vs invoice `created_at` precision. Use `NOW()` comparison in trigger.

### 4.5 Lead Attribution Permanent

**Rule**: ÔÇťAttribution is permanent ÔÇô reassignment requires admin override with audit trail.ÔÇŁ (rule 7)

**Implementation**: `channel_partner_leads.lead_user_id` UNIQUE constraint prevents multiple ChanPs per lead. Admin reassignment requires deleting old record? Actually admin would need to update `channel_partner_id` (or delete and insert) with audit log.

**Contradiction?** UNIQUE constraint prevents duplicate attribution but does not prevent admin from updating `channel_partner_id`. Should we allow update? Spec says ÔÇťreassignment requires admin overrideÔÇŁ. Implementation should allow admin to update `channel_partner_id` with audit log. Ensure RLS allows admin update.

### 4.6 Dual Roles Restriction

**Rule**: ÔÇťA user cannot simultaneously be a ChanP and a lead of another ChanP. A ChanP is not eligible to be attributed as a lead.ÔÇŁ (rule 10)

**Implementation**: Need validation:
- On ChanP registration, ensure user does not already have a `channel_partner_leads` record.
- On lead attribution, ensure lead user is not a ChanP (check `profiles.active_role` or `channel_partners` existence).
- Database constraint? Could add CHECK that `lead_user_id` not in `channel_partners.user_id`. Use trigger.

**Recommendation**: Add trigger `before insert or update on channel_partner_leads` to reject if lead user is a ChanP.

### 4.7 Lead Role Determination Ambiguity

**Rule**: ÔÇťlead_role: Lead role at invoice time (determines which rate was used)ÔÇŁ (spec 4.4).

**Ambiguity**: Should `lead_role` be taken from `channel_partner_leads.lead_role` (role at attribution time) or from the invoiceÔÇÖs `from_party_role` (role at invoice time)? The spec states ÔÇťLead role at invoice timeÔÇŁ, which suggests the latter. However, `channel_partner_leads.lead_role` is the role at attribution time and may differ if the lead changes roles later.

**Implications**:
- If using `from_party_role`, the commission rate may differ from the rate configured for the attribution role. This aligns with ÔÇťper-role configurable rateÔÇŁ ÔÇô the rate applies to the role the lead is acting as when generating income.
- If using `channel_partner_leads.lead_role`, the commission rate remains fixed based on initial attribution, which could be incorrect if the lead switches roles.

**Recommendation**: Use `from_party_role` as `lead_role` for commission calculation. Store this value in `channel_partner_commissions.lead_role`. Ensure that `from_party_role` is an eligible lead role (guardian, agency, caregiver, shop). If not, skip commission.

**Validation**: Update commission trigger logic to join `channel_partner_leads` on `lead_user_id = from_party_id` and use `from_party_role` for rate lookup.

### 4.8 Commission Calculation Rounding

**Requirement**: Commission amount must be integer CarePoints (no fractional points). Calculation: `platform_fee * (cp_commission_rate / 100)`.

**Ambiguity**: Rounding method not specified. Should be rounded to nearest integer (half up). Use PostgreSQL `ROUND()`.

**Edge Cases**:
- Platform fee zero Ôćĺ commission zero.
- Rate zero Ôćĺ commission zero.
- Very small platform fee (e.g., 1 CarePoint) with rate 1% Ôćĺ 0.01 rounded to 0 (no commission). Acceptable.

**Validation**: Ensure `cp_commission_amount` column is INTEGER and rounded value stored. Use `ROUND(platform_fee * cp_commission_rate / 100)`.
