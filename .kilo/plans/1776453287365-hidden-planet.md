# Fix: Validate Patient Query Against Real Supabase

## Problem

The caregiver chat feature's patient-user query (Step 3b in `caregiverContacts.service.ts`) has never been executed against a real Supabase instance. All tests run in mock mode. The 2-step query is structurally correct based on schema analysis, but could still fail due to RLS policies, missing data, or PostgREST quirks.

A standalone test script (`test-patient-query-node.js`) was created but never committed to version control properly, and the credentials were hardcoded. The proper approach is a Vitest integration test that auto-skips when env vars are absent (matching the existing pattern in `lifecycle.db.integration.test.ts`).

## What Exists

- `.env` has real Supabase URL + anon key
- `.env` has `TEST_GUARDIAN_EMAIL` / `TEST_GUARDIAN_PASSWORD` (guardian account)
- `lifecycle.db.integration.test.ts` — established pattern for real Supabase integration tests
- `test-patient-query-node.js` — standalone script with hardcoded caregiver credentials (not a proper test)

### 2. Create proper Vitest integration test

**File**: `src/backend/services/__tests__/caregiverContacts.integration.test.ts`

Follow the exact pattern from `lifecycle.db.integration.test.ts`:
- Import `createClient` directly from `@supabase/supabase-js` (bypasses `supabase.ts` which requires Vite)
- Read env vars from `process.env` (works in Node.js)
- `describe.skipIf(!run)` — auto-skips when env vars are absent
- Authenticate with `signInWithPassword`
- Execute each query from the service against real Supabase
- Assert results are valid (structure, not necessarily specific data)

```typescript
import { afterAll, describe, expect, it } from "vitest";
import { createClient, type SupabaseClient } from "@supabase/supabase-js";

const URL = process.env.VITE_SUPABASE_URL;
const ANON = process.env.VITE_SUPABASE_ANON_KEY;
const EMAIL = process.env.TEST_CAREGIVER_EMAIL;
const PASSWORD = process.env.TEST_CAREGIVER_PASSWORD;
const run = Boolean(URL && ANON && EMAIL && PASSWORD);

let sb: SupabaseClient | null = null;
let caregiverId: string | null = null;

async function getClient(): Promise<SupabaseClient> {
  if (!sb) {
    sb = createClient(URL!, ANON!);
    const { data, error } = await sb.auth.signInWithPassword({ email: EMAIL!, password: PASSWORD! });
    if (error) throw error;
    caregiverId = data.user.id;
  }
  return sb;
}

describe.skipIf(!run)("caregiverContacts (real Supabase)", () => {
  afterAll(async () => {
    if (sb) await sb.auth.signOut();

  it("Step 1: gets CAC IDs for caregiver", async () => {
```

### 4. Run the test


## Possible Outcomes

1. **All queries pass** — Feature is validated. Close the issue.
2. **RLS blocks a query** — Need to check if the test caregiver has the right data (active placements, etc.), or if RLS policies need adjustment.
3. **Empty results but no errors** — The test caregiver may not have active jobs/patients in the DB. Need to seed test data or use a different test account.
4. **PostgREST error** — Query structure issue. Fix and re-test.
