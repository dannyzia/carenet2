# Fix: Broken Profile Navigation for Gated Users

## Problem

Two bugs in `CompleteProfilePage.tsx`:

1. **Wrong routes** — `getProfileLink()` returns `/agency/profile` and `/shop/profile`, which don't exist in `routes.ts`. The correct routes are `/agency/storefront` and `/shop/onboarding`.

2. **Redirect loop** — Even the working caregiver link (`/caregiver/profile`) is unreachable. All profile pages live under `ProtectedRoute`, which gates users with `activationStatus === 'profile_incomplete'` back to `/auth/complete-profile`. Clicking "Go to Profile" → ProtectedRoute gates → redirects back to holding page. This affects **all** gated roles, not just agency/shop.

## Fix

### 1. Add gate exemption for profile routes in `ProtectedRoute.tsx`

Add a list of paths that `profile_incomplete` users are allowed to visit. When a gated user navigates to their own role's profile page, let them through.

```ts
const PROFILE_EDIT_PATHS: Record<string, string[]> = {
  caregiver: ['/caregiver/profile'],
  agency: ['/agency/storefront'],
  shop: ['/shop/onboarding'],
};
```

In the gate block, before the `switch (status)` redirect, check if the current path is an allowed profile-edit path for this role:

```ts
if (status === 'profile_incomplete') {
  const allowed = PROFILE_EDIT_PATHS[user.activeRole] || [];
  if (allowed.some(p => location.pathname.startsWith(p))) {
    return <Outlet />;
  }
  return <Navigate to="/auth/complete-profile" replace />;
}
```

This way, a caregiver with `profile_incomplete` can access `/caregiver/profile` to edit their details, but gets gated on all other protected routes.

### 2. Fix route paths in `CompleteProfilePage.tsx`

Update `getProfileLink()` to return actual registered routes:

```ts
case "caregiver": return "/caregiver/profile";    // ✅ already correct
case "agency":    return "/agency/storefront";     // was /agency/profile
case "shop":      return "/shop/onboarding";       // was /shop/profile
```

## Files Changed

| File | Change |
|---|---|
| `src/frontend/components/guards/ProtectedRoute.tsx` | Add `PROFILE_EDIT_PATHS` map + gate exemption for `profile_incomplete` users visiting their own profile route |
| `src/frontend/pages/auth/CompleteProfilePage.tsx` | Fix `getProfileLink()` return values for agency and shop |

## Risk

- **Low.** The exemption only applies when `status === 'profile_incomplete'` AND the path matches the user's own role's profile route. All other gated statuses (`pending_approval`, `rejected`, `suspended`) still gate fully. The exemption is additive — it only opens paths that were previously inaccessible, never restricts existing access.
