# Plan: Fix Demo Access Role Pages — Long-Term Solution

## Problem Summary

Two systemic issues prevent demo access role pages from working correctly:

1. **Broken Channel Partner navigation**: The role `"channel_partner"` does not map to its route prefix `/cp/`. Every location that interpolates the role string directly into a URL (e.g. `` `/${role}/dashboard` ``) produces `/channel_partner/dashboard`, which is a 404. There is no centralized mapping; the role-to-prefix relationship is scattered across 20+ files.

2. **Dashboard crash on 5 of 8 roles**: The `getOperationalDashboard()` methods in caregiver, guardian, patient, agency, and shop services have no try-catch fallbacks. When `USE_SUPABASE` is true but the demo user has no real Supabase auth session, the REST API returns 401 Unauthorized. This uncaught error propagates through `useAsyncData` and renders "Something went wrong." Admin and moderator services already follow the correct pattern (try-catch with fallback).

## Part A: Centralized Role-to-Path Mapping

### A1. Create `src/backend/navigation/roleAppPaths.ts`

Create a new module that serves as the single source of truth for mapping any `Role` to its route prefix and canonical dashboard path.

```
File: src/backend/navigation/roleAppPaths.ts
```

**What it must export:**

1. **`ROLE_ROUTE_PREFIX: Record<Role, string>`** — A complete mapping from every `Role` to its URL prefix:
   - `"caregiver"` → `"/caregiver"`
   - `"guardian"` → `"/guardian"`
   - `"patient"` → `"/patient"`
   - `"agency"` → `"/agency"`
   - `"admin"` → `"/admin"`
   - `"moderator"` → `"/moderator"`
   - `"shop"` → `"/shop"`
   - `"channel_partner"` → `"/cp"`

   The `Role` type is imported from `@/frontend/auth/types`. The mapping is `as const` so TypeScript enforces completeness — if a new role is added to the union, this object must be updated or the build fails.

2. **`roleDashboardPath(role: Role): string`** — Returns the full dashboard URL for a role. Example: `roleDashboardPath("channel_partner")` returns `"/cp/dashboard"`. Implementation: `return \`${ROLE_ROUTE_PREFIX[role]}/dashboard\``.

3. **`roleMessagesPath(role: Role): string`** — Returns `"/cp/messages"` for channel_partner, `"/caregiver/messages"` for caregiver, etc.

4. **`roleMarketplaceHubPath(role: Role): string`** — Returns the marketplace-hub path for care-seeker roles (guardian, patient, caregiver). For roles without a marketplace hub, returns the dashboard path.

5. **`rolePath(role: Role, suffix: string): string`** — Generic path builder. Returns `\`${ROLE_ROUTE_PREFIX[role]}/${suffix}\`` (suffix should NOT start with `/`). This covers any future path construction need.

**Design rules for this file:**
- Import `Role` from `@/frontend/auth/types`.
- No other imports needed.
- Every function is a pure function with no side effects.
- The file must be importable from both frontend and backend code (no DOM or React dependencies).

### A2. Create `src/backend/navigation/pathToRole.ts`

Create a companion module that does the reverse mapping (URL segment → Role), needed by `BottomNav`, `AuthenticatedLayout`, and `PublicNavBar`.

```
File: src/backend/navigation/pathToRole.ts
```

**What it must export:**

1. **`ROUTE_PREFIX_TO_ROLE: Record<string, Role>`** — The inverse of `ROLE_ROUTE_PREFIX`. Maps URL segments like `"cp"` → `"channel_partner"`, `"caregiver"` → `"caregiver"`, etc. Build this programmatically from `ROLE_ROUTE_PREFIX` (iterate entries, strip the leading `/`, create reverse map).

2. **`roleFromPath(pathname: string): Role | null`** — Extracts the first path segment from `pathname` (after the leading `/`), looks it up in `ROUTE_PREFIX_TO_ROLE`, and returns the matching `Role` or `null` if none matches.

### A3. Update all navigation call sites

Every file listed below must be updated to use the new utility instead of inline role interpolation. The change pattern is the same everywhere: replace `` `/${role}/dashboard` `` with `roleDashboardPath(role)`, and `` `/${role}/messages` `` with `roleMessagesPath(role)`, etc.

**Import pattern for all updated files:**
```ts

```

### B2. Create a shared `withDashboardFallback` wrapper

Create a small utility function in a new file:

```
File: src/backend/navigation/withDashboardFallback.ts
```

No — this belongs in the service layer, not navigation. Better location:

```
File: src/backend/services/dashboardFallback.ts
```

**What it must export:**

1. **`withDashboardFallback<T>(fn: () => Promise<T>, emptyFallback: T, mockPicker?: (m: MockBarrel) => T): Promise<T>`**

   A generic wrapper that:
   - Calls `fn()` inside a try-catch
   - On success, returns the result
   - On error, logs a warning (using `console.warn`, matching the admin service pattern)
   - If `useInAppMockDataset()` is true and `mockPicker` is provided, loads the mock barrel and returns `mockPicker(mockBarrel)`
   - Otherwise, returns `emptyFallback`

   This encapsulates the exact pattern already used by `fetchAdminDashboardLegacy` and moderator's `getDashboardStats`/`getDashboardQueue`, but makes it reusable.

2. **`EMPTY_OPERATIONAL_DASHBOARD`** — Re-exported from `liveEmptyDefaults.ts` for convenience.

**Import pattern:**
```ts
import { useInAppMockDataset } from "./_sb";
import { loadMockBarrel, type MockBarrel } from "./demoOfflineMock";
```

**Implementation sketch:**
```ts
export async function withDashboardFallback<T>(
  fn: () => Promise<T>,
  emptyFallback: T,
  mockPicker?: (m: MockBarrel) => T,
): Promise<T> {
  try {
    return await fn();
  } catch (error) {
    console.warn("[CareNet Service] Supabase call failed, falling back:", error);
    if (mockPicker && useInAppMockDataset()) {
      const m = await loadMockBarrel();
      return mockPicker(m);
    }
    return emptyFallback;
  }
}
```
